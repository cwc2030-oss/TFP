### Saddle Quality Filter — Fewer, Defensible Saddles (2026-05-02)
- **Root cause**: `filterSpinesByQuality()` only filtered RIDGES — saddle_nodes passed through completely unfiltered from Modal AND synthetic pipeline
- **NEW FUNCTION**: `filterSaddlesByQuality()` in `lib/ridge-extraction.ts` — 5-stage pipeline:
  1. **Prominence filter**: ridgeDropFt ≥ 20ft (SADDLE_MIN_PROMINENCE_FT)
  2. **Ridge proximity filter**: must be within 150m of a ridge spine (SADDLE_MAX_DIST_FROM_RIDGE_M)
  3. **Confidence filter**: ≥ 0.3 (SADDLE_MIN_CONFIDENCE) — for Modal-provided confidence scores
  4. **Spacing filter**: greedy-pick best saddles first, enforce 300m minimum spacing (SADDLE_MIN_SPACING_M)
  5. **Density cap**: max 1.5 saddles per km of total ridge length (SADDLE_MAX_PER_KM_RIDGE)
- Sort score for greedy pick: 60% prominence + 30% proximity + 10% confidence
- Applied in BOTH `fetchRidgeSpines` client path AND `/api/ridge-spines` server route
- Returns `SaddleDebugPayload` exported type with full per-candidate diagnostics
- `[SaddleDebug]` console emission in intel/page.tsx: colored header, full JSON, console.table of all candidates
- Server-side `terrain_debug.pipeline_steps.saddle_filter` shows raw → final counts + all stage counts
- Helper: `pointToSegmentDistM()` — perpendicular distance from point to line segment for ridge proximity

### v4.0 Cache-Based Territory Assembly (2026-05-11)
Rebuilt Territory mode from "re-analyze merged polygon" to "assemble from per-parcel cache."

#### Architecture:
- **TerrainAnalysisCache** Prisma model: stores per-parcel terrain results (layers, tieredCorridorData, ridgeSpineData, terrainFlowData, summary) keyed by parcelId with 7-day TTL
- **API**: `/api/terrain-cache` — GET (batch read by parcelIds), POST (upsert single parcel)
- **Cache Write**: useEffect watches layers+tieredCorridorData+ridgeSpineData+terrainFlowData+summary. When all are non-null in single-parcel mode, fires a fire-and-forget write to cache.
- **Territory Assembly** (`lib/territory-assembly.ts`):
  - `assembleTerritory()`: Merges per-parcel FeatureCollections, tags each feature with `_sourceParcelId`
  - `computeCrossParcelLinks()`: For adjacent parcel pairs (300m threshold), finds best corridor/ridge handoff endpoints and generates connecting LineStrings
  - Exports `fetchCachedTerrain()` and `writeCachedTerrain()` helpers

#### Analyze Territory / Re-Align button (intel/page.tsx):
1. Fetch cached terrain for all territory parcels via `/api/terrain-cache`
2. For cache misses: run individual terrain analysis (corridors + ridges + flow) per parcel, cache results
3. Assemble: merge all FCs, compute cross-parcel links, **aggregate metadata** (feature counts, total lengths, dem_source)
4. **Step 3b**: For territories with >1 parcel, compute unified territory-wide deer flow via single `fetchTerrainFlow()` call with merged MultiPolygon. Override `assembled.terrainFlowData` with unified result. Cache with SHA-256 territory hash key.
5. Inject directly into state (layers, tieredCorridorData, ridgeSpineData, terrainFlowData, summary, territoryLinks)
6. `territoryAssemblyRef` gate prevents ridge/flow/tiering useEffects from re-analyzing the merged polygon

#### Polyline Smoothing (lib/polyline-smooth.ts)
- Ridge spine polylines are smoothed display-only before pushing to Mapbox sources
- Pipeline: Douglas-Peucker simplification (0.00005° tolerance ≈ 5m) → Chaikin corner-cutting (2 passes)
- Applied in intel/page.tsx at setData() calls for tfp-ridges-primary and tfp-ridges-secondary sources
- Raw data in React state is untouched — only visual rendering is smoothed
- Downstream consumers (hunt pockets, stand directions, anchors) use original coordinates

## Re-Align Territory Progress Counter Fix (2026-05-29)
- **Root cause**: Progress counter used `cacheResult.missing.length` as denominator — only cache misses, not total parcels
- **Symptom**: First analysis showed correct N/M (all misses = all parcels), Re-Align showed wrong counts (cached parcels excluded from denominator)
- **Secondary bug**: Cache API double-counted corrupt entries in `missing` array (pushed in catch block AND in "not found" loop)
- **Fix**: Counter now uses `cachedCount + mi + 1` of `totalParcels` (parcels.length) — always reflects real territory size
- **Defense**: Deduplicate parcelIds with Set before cache lookup; cache API uses Set-based tracking to prevent inflation
- **Progress messages**: "All N parcels found in cache — assembling..." / "X of N parcels cached — analyzing Y remaining..." / "Analyzing parcel X of N..."
