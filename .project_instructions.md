## Single-parcel deer flow "wake up" fix (hero experience)
Problem: a single parcel opened lifeless (sparse flow, ~2 stands) because deer flow was computed over a ~1km buffered extent then clipped back TIGHT to the parcel boundary — discarding all off-property corridors that enter/cross/converge. Flow is the hero of the product; it must never open dead.

Fixes applied:
- `lib/terrain-flow.ts` `generateTerrainDrivenFlow`: for a SINGLE parcel (`!isTerritory`), flow lines / convergence / opportunity zones are now clipped against an 800m buffered hunt-context ring (`FLOW_CONTEXT_BUFFER_M = 800`, built via `createBufferedParcel`) instead of the tight parcel. Single-parcel `overlapThreshold` lowered 0.40 → 0.25 (territory stays 0.20). Has try/catch fallback to tight parcel rings.
- `lib/terrain-analysis.ts` `computeParcelScale`: added `lengthScale = 1 + (scaleFactor-1)*0.5` and switched `minLengthPrimary/minLengthSecondary` from `*scaleFactor` to `*lengthScale`. Full 2.5x scale had pushed primary min-length to ~375m on ~300-acre parcels, silently dropping strong-but-shorter corridors (`extractFlowLines` DROPS lines below the tier min). Now ~263m/140m at 2.5x. Small parcels (scaleFactor 1.0) unaffected; territory still uses min 0.

GUARDRAIL (critical): the 800m buffer governs ONLY the flow visualization. Stands, the report, and all parcel data stay scoped to the selected parcel. Neighbor influence shows via flow only; neighbor stands/boundaries/owner data/reports stay locked. Stand count naturally improves because `anchorConvergencePoints` (a stand-anchor source in `app/intel/page.tsx`) is fed by `terrainFlowData.convergence_zones`, which now extends into the ring — no dishonest padding was added (user values honest stand counts; existing TOP-UP caps at anchored candidates).

## Stand-Alignment Engine v1.1 — Inside-Corner Factor (added)

The per-stand alignment score lives in `lib/scoring/stand-alignment.ts` (`computeStandScore`), NOT in the parcel-level `weights/v1_0.yaml`. v1.1 weights (`STAND_ALIGNMENT_WEIGHTS_V1_1`, documented in `weights/stand-alignment-v1_1.yaml`): wind 0.35 (king), inside_corner 0.20 (new), movement 0.17 (was 0.25), intrusion 0.20 (was 0.25), time 0.05, season 0.03 — sum 1.00. The 0.20 was pulled mostly from movement on purpose (a corner is itself a movement signal → avoids structural triple-dipping).

`StandInputs.inside_corner?: number | null` is tri-state: a number = parcel HAS corners, factor applies (0 for a stand far from any corner); null/undefined = parcel has NO corners → ADAPTIVE RENORMALIZATION GUARD redistributes the 0.20 proportionally across the other 5 factors (÷0.80) so timber parcels are never penalized. Generosity curve `pow(raw,0.85)` and `round(*100)` preserved.

Wiring: `computeInsideCornerScore()` (module fn in `app/intel/page.tsx`) = distance decay (~70m falloff) × sharpness ((160-interiorAngle)/100) × corridor-mouth boost (≤60m → up to ×1.25). Corners come from CDL `findInsideCorners()` via `cdlCornersRef` (synced in an effect on `cdlData.insideCorners`, which also re-fires `computeAlignmentScores`). Reason chip "🎯 Inside Corner" added in `stand-explainability.ts` (getInsideCornerChip). Debug tool `stand-selection-debug.ts` updated to v1.1 weights + inside_corner_contribution.

Verified numerically (VM can't capture Mapbox visuals): farm parcel — stand IN a sharp corridor-fed corner 75→81, stands far from corners drop (snap-to-corner); timber parcel — scores essentially unchanged (+1..+3), guard confirmed not cratering.

## Deer-flow large-territory coverage fix (v4.2)

Problem: On large territories (~100 parcels / ~10k acres), outer/southern parcels rendered stand markers but NO flow lines/corridors/zones, even at 100/100 analysis. Root causes found in the real-DEM flow path (`/api/terrain-flow` -> `generateTerrainDrivenFlow` in `lib/terrain-flow.ts`):
1. `computeTerrainConvergence` (lib/terrain-analysis.ts) normalizes corridor density by a single GLOBAL maxCount -> southern (sparse) regions get near-0 likelihood -> their corridors fall below `secondary_percentile` (0.55) in `extractFlowLines` and are culled.
2. `computeParcelScale` zone budgets were hard global ceilings (`Math.min(10,...)`/`Math.min(6,...)`), giving a 10k-acre territory the same ~6 conv/6 opp zones as a small one, all clustered centrally.

Fix (all territory-gated via `isTerritory`; single-parcel & small-territory output byte-identical — verified):
- `computeParcelScale`: added `isTerritory` to `ParcelScaleMetrics`; in territory mode `maxConvergenceZones`/`maxOpportunityZones` scale with AREA (4 / 2.5 per 1000 acres) with floor 6 (== prior territory value) and perf ceilings 48 / 30.
- `extractFlowLines`: added optional `FlowBinningOptions` (4th arg). Territory caller passes `{ bbox: bufferedBbox }`. It runs an ADDITIVE per-bin pass (coarse grid, max 12x12 bins, top-3 per bin, scoreFloor 0.30) that adds each region's strongest corridors as `secondary` flow lines even if below the global threshold — guarantees southern coverage. Non-territory passes `undefined` (unchanged).
- Debug ridge/saddle slice caps in terrain-flow.ts (was 50/30) now sqrt(area)-scaled in territory mode only.
- v3 synthetic ceilings (terrain-flow-v3.ts numPrimary/numSecondary/numFunnels/numCrossings/MAX_ZONES) scale up in territory mode via `scale.isTerritory` (fallback path only).

Verification: unit-tested the real compiled `extractFlowLines`/`computeParcelScale` with synthetic depressed-south data — southern flow lines went 0 -> 8, single-parcel byte-identical. A live 100-parcel territory (needs Modal + real completed analysis in DB) was NOT reproduced in-browser; report this honestly.
Left intentionally unchanged: `rescoreStandSites` slice(0,3) in terrain-heatmap.ts is a deliberate "top-3 stand recommendations" product cap, not the flow bug.

## Deer-flow territory threshold — empirical fingerprint (founder-observed)

Founder (Clark) observed empirically that **flow stops extending south at ~70 parcels / ~7,000 acres** on large territories. This is the exact fingerprint of the old hard 50-ridge / 30-saddle feature budget: at ~70 parcels the 80 feature slots saturate with the strongest (northern, high-convergence) features, so parcels added south of there get zero features and flow can't cross the line.

Use this as the regression test target for the v4.2 territory-scale fix: a 90–100 parcel territory should render flow edge-to-edge (including southern parcels), NOT pile up in the northern band. If flow still stops short of the south on a large completed territory, the fix has regressed.

## Product principle — low-scoring land framing ("winners and losers")

Terrain-graded deer data prices land, so the Terrain Brain crowns winners AND stamps losers. A landowner whose ground scores low is still a landowner we want listing/on-platform. Framing of the low end matters: lead with "not a premium whitetail funnel, but here's what it IS good for" (alternative uses/species/value) rather than "your land is weak" — the former retains the owner, the latter sends them to a competitor. Apply this whenever surfacing low flow scores / low grades in UI, PDFs, or emails. (Not an urgent build item — a framing guardrail to honor in any low-score-facing copy.)

## Deer-flow large-territory convergence fix (v4.3)

Follow-up to v4.2. Founder reported: consistently at ~30 parcels / ~3,000 acres, convergence zones balloon and merge into a single broad band spanning 5,000+ acres, and blue (moderate) flow trends north.

Root cause (confirmed): `identifyConvergenceZones` (lib/terrain-analysis.ts) computed zone radius as `baseRadius + flowCount*(baseRadius/3)` with **unbounded** flowCount (it counts every flow within searchRadius). On a dense basin flowCount hits dozens → km-scale radii that visually merge into one band. It was the outlier — the v3 (terrain-flow-v3.ts:945) and terrain-flow.ts:1122 variants already clamp flowCount to min(4,...). Also zone SELECTION was global top-N maxima (`maxima.slice(0,maxZones)`), so all zones landed in the dominant central/northern density band.

Fix (all gated behind `isLargeTerritory = isTerritory && areaAcres >= 3000` so single parcels and <3,000ac territories are byte-identical):
- `identifyConvergenceZones`: clamp radius flowCount to 4 AND hard-cap final radius at `CONVERGENCE_MAX_RADIUS_M = 120`m. Convergence must render as a local pinch, never a multi-mile band.
- `identifyConvergenceZones`: replaced global top-N maxima selection with `selectSpatiallyDistributedMaxima()` — buckets maxima into a coarse grid (<=12x12 bins, 900m target) and picks each bin's strongest in rounds until maxZones filled. Deterministic. Distributes zones across the extent (incl. south) instead of piling up north.
- `ZoneScalingOptions` gained `isLargeTerritory?` and `bbox?`; wired from terrain-flow.ts zoneScaling (`bbox: bufferedBbox`).
- terrain-flow-v3.ts:945 raw flowCount clamped to 4 for large territories too.

Verified with unit tests against the REAL compiled `identifyConvergenceZones`: OLD maxRadius 315m → NEW 105m; with scarce slots (maxZones=6) OLD gave 0 southern zones, NEW gives 2; isLargeTerritory:false output byte-identical to pre-change. tsc clean.

North-bias (cause #2): per founder guidance, fixed the radius/distribution first (which itself reduces the northward density pull since zones no longer all cluster north). Whether a residual directional bias remains in corridor endpoint selection / Modal pathfinding is to be RE-CHECKED on a live large territory before any further change.
