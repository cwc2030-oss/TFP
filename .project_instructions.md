## Single-parcel deer flow "wake up" fix (hero experience)
Problem: a single parcel opened lifeless (sparse flow, ~2 stands) because deer flow was computed over a ~1km buffered extent then clipped back TIGHT to the parcel boundary тАФ discarding all off-property corridors that enter/cross/converge. Flow is the hero of the product; it must never open dead.

Fixes applied:
- `lib/terrain-flow.ts` `generateTerrainDrivenFlow`: for a SINGLE parcel (`!isTerritory`), flow lines / convergence / opportunity zones are now clipped against an 800m buffered hunt-context ring (`FLOW_CONTEXT_BUFFER_M = 800`, built via `createBufferedParcel`) instead of the tight parcel. Single-parcel `overlapThreshold` lowered 0.40 тЖТ 0.25 (territory stays 0.20). Has try/catch fallback to tight parcel rings.
- `lib/terrain-analysis.ts` `computeParcelScale`: added `lengthScale = 1 + (scaleFactor-1)*0.5` and switched `minLengthPrimary/minLengthSecondary` from `*scaleFactor` to `*lengthScale`. Full 2.5x scale had pushed primary min-length to ~375m on ~300-acre parcels, silently dropping strong-but-shorter corridors (`extractFlowLines` DROPS lines below the tier min). Now ~263m/140m at 2.5x. Small parcels (scaleFactor 1.0) unaffected; territory still uses min 0.

GUARDRAIL (critical): the 800m buffer governs ONLY the flow visualization. Stands, the report, and all parcel data stay scoped to the selected parcel. Neighbor influence shows via flow only; neighbor stands/boundaries/owner data/reports stay locked. Stand count naturally improves because `anchorConvergencePoints` (a stand-anchor source in `app/intel/page.tsx`) is fed by `terrainFlowData.convergence_zones`, which now extends into the ring тАФ no dishonest padding was added (user values honest stand counts; existing TOP-UP caps at anchored candidates).
